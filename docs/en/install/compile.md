If your compilation environment can access network, please refer to [Online Compilation](#online-compilation). Compilation script will automatically download third-party software online. Otherwise, please refer to [Offline Compilation](#offline-compilation) to manually download third-party software.

After preparing open source third-party software, can use the following compilation methods. Please choose as needed:

- **Custom Operator Package**:

  Package generated by selecting partial operators for compilation is called custom operator package, acts on CANN package in **mounting** form, does not change original package content. Generated custom operator package has higher priority than original CANN package. This package supports aclnn and graph mode invocation of AI Core, AI CPU operators.

- **ops-math Package**:

  Package generated by compiling entire project is called ops-math package, can **completely replace** corresponding part of CANN package. This package supports aclnn and graph mode invocation of AI Core operators.

- **ops-math Static Library**:

  > Note: If you need to **perform secondary release based on this project** and have requirements for **software package size**, suggest using static library compilation. This library can link your application development program, only retaining operators needed by business, thereby achieving software minimized deployment.

  Means compiling entire project into one static library file, including libcann_math_static.a and aclnn interface header files. This package only supports aclnn invocation of AI Core operators.

## Online Compilation

If compiling in environment with internet, third-party dependencies will be automatically installed during compilation process. No manual installation required. Source code compilation and deployment commands in different scenarios are as follows:

### Scenarios Without Using Spack

#### Custom Operator Package

1. **Compile Custom Operator Package**

    Enter project root directory, execute the following compilation command:

    ```bash
    bash build.sh --pkg --soc=${soc_version} [--vendor_name=${vendor_name}] [--ops=${op_list}] [-j${n}]
    # Taking Abs operator compilation as example
    # bash build.sh --pkg --soc=ascend910b --ops=abs -j16
    # Compile user operators under experimental contribution directory (taking Abs operator as example, please use actual contributed operator during compilation)
    # bash build.sh --pkg --experimental --soc=ascend910b --ops=abs -j16
    ```

    - --soc: $\$\{soc\_version\}$ represents NPU model. Atlas A2 training series products/Atlas A2 inference series products use "ascend910b" (default), Atlas A3 training series products/Atlas A3 inference series products use "ascend910_93", Ascend 950PR/Ascend 950DT products use "ascend950".
    - --vendor_name (optional): $\$\{vendor\_name\}$ represents built custom operator package name, default name is custom.
    - --ops (optional): $\$\{op\_list\}$ represents operators to compile, default compiles all operators when not specified. Format like "abs,add_lora,...", multiple operators separated by English comma ",".
    - --experimental (optional): Represents compiling operators saved by user under experimental contribution directory.
    - -j (optional): Specify compilation thread count to speed up compilation.

    If neither $\$\{vendor\_name\}$ nor $\$\{op\_list\}$ is passed in, ops-math package is compiled; if compiling custom operator package of all operators, need to pass in $\$\{vendor\_name\}$. When following message appears, compilation is successful.

    ```bash
    Self-extractable archive "cann-ops-math-${vendor_name}_linux-${arch}.run" successfully created.
    ```

    After successful compilation, run package is stored in build_out directory under project root directory.

2. **Install Custom Operator Package**

    ```bash
    ./build_out/cann-ops-math-${vendor_name}_linux-${arch}.run
    ```

    Custom operator package default installation path is ```${ASCEND_HOME_PATH}/opp/vendors```. $\$\{ASCEND\_HOME\_PATH\}$ has been configured through environment variable, represents CANN toolkit package installation path, generally $\$\{install\_path\}/cann.

    > Note: Can define operator package installation directory by configuring --install-path=$\$\{install\_path\} parameter. Before using custom operator package, need to execute source $\$\{install\_path\}/vendors/${vendor_name}/bin/set_env.bash command. set\_env.bash script appends custom operator package installation path to environment variable ASCEND\_CUSTOM\_OPP\_PATH, making custom operator package effective in current environment.

3. **(Optional) Uninstall Custom Operator Package.**

    After custom operator package installation, `uninstall.sh` will be generated in ```${ASCEND_HOME_PATH}/opp/vendors/${vendor_name}_math/scripts``` directory. Can uninstall custom operator package through this script. Command as follows:

    ```bash
    bash ${ASCEND_HOME_PATH}/opp/vendors/${vendor_name}_math/scripts/uninstall.sh
    ```

#### ops-math Package

> **Note**: Full compilation will build all operators in repository, taking longer time, related to machine configuration, `-j` parallelism and whether to download third-party dependencies online; for daily development, please prioritize using single operator compilation (`--ops=<operator_name>`).

1. **Compile ops-math Package**

    Enter project root directory, execute the following compilation command:

    ```bash
    # Compile all operators except experimental directory
    bash build.sh --pkg --soc=${soc_version} [-j${n}]
    # Such as: bash build.sh --pkg --soc=ascend910b -j16
    # Compile all operators under experimental directory
    # bash build.sh --pkg --experimental --soc=${soc_version} [-j${n}]
    ```

    - --soc: $\$\{soc\_version\}$ represents NPU model. Atlas A2 training series products/Atlas A2 inference series products use "ascend910b" (default), Atlas A3 training series products/Atlas A3 inference series products use "ascend910_93", Ascend 950PR/Ascend 950DT products use "ascend950".
    - --experimental (optional): Represents compiling operators saved by user under experimental directory.
    - -j (optional): Specify compilation thread count to speed up compilation.

    If following message appears, compilation is successful.

    ```bash
    Self-extractable archive "cann-${soc_name}-ops-math_${cann_version}_linux-${arch}.run" successfully created.
    ```

    $\$\{soc\_name\}$ represents NPU model name, i.e., content remaining after removing "ascend" from $\$\{soc\_version\}$. After successful compilation, run package is stored in build_out directory.

2. **Install ops-math Package**

    ```bash
    # Installation command
    ./build_out/cann-${soc_name}-ops-math_${cann_version}_linux-${arch}.run --full --install-path=${install_path}
    ```

    $\$\{install\_path\}$: Represents specified installation path, needs to be installed in same path as toolkit package, default installed in `/usr/local/Ascend` directory.

3. **(Optional) Uninstall ops-math Package**

    ```bash
    # Uninstall command
    ./${install_path}/cann/share/info/ops_math/script/uninstall.sh
    ```

#### ops-math Static Library

> Note: Static library only supports Atlas A2, Atlas A3 series products. experimental operators do not support using static library temporarily.

1. **Compile ops-math Static Library**

    Enter project root directory, execute the following compilation command:

    ```bash
    bash build.sh --pkg --static --soc=${soc_version} [-j${n}]
    ```

    - --soc: $\$\{soc\_version\}$ represents NPU model. Atlas A2 series products use "ascend910b" (default), Atlas A3 series products use "ascend910_93".
    - -j (optional): Specify compilation thread count to speed up compilation.
    If following message appears, compilation and compression are successful.

    ```bash
    [SUCCESS] Build static lib success!
    Successfully created compressed package: ${repo_path}/build_out/cann-${soc_name}-ops-math-static_${cann_version}_linux-${arch}.tar.gz
    ```

    $\$\{repo\_path\}$ represents project root directory, $\$\{soc\_name\}$ represents NPU model name, i.e., content remaining after removing "ascend" from $\$\{soc\_version\}$. After successful compilation, compressed package is stored in build_out directory.

2. **Extract ops-math Static Library**

    Enter build_out directory and execute extraction command:

    ```bash
    tar -zxvf ./cann-${soc_name}-ops-math-static_${cann_version}_linux-${arch}.tar.gz -C ${static_lib_path}
    ```

    $\$\{static\_lib\_path\}$ represents static library extraction path. Directory structure after extraction is as follows:

    ```text
    ├── cann-${soc_name}-ops-math-static_${cann_version}_linux-${arch}
    │   ├── lib64
    │   │   ├── libcann_math_static.a               # Static library file
    │   └── include
    |       ├── ...                                 # aclnn interface header files
    ```

## Offline Compilation

If compiling in environment without internet connection, need to prepare dependent third-party software in advance, then perform source code compilation. Specific process is as follows:

1. **Download Third-Party Dependencies**

    Download third-party software in advance in networked environment. Currently there are the following methods, please choose as needed:

    - Method 1: Manually download according to table provided in [Third-Party Software Dependencies](#install-third-party-dependencies). If downloading from other addresses, please ensure version numbers are consistent.

    - Method 2: One-click download through [third_lib_download.py](../../../scripts/tools/third_lib_download.py) script. This script is in project `scripts/tools/` directory. Download this script and execute the following command:

        ```bash
        python ${scripts_dir}/third_lib_download.py
        ```

    $\$\{scripts\_dir\}$ represents script storage path. Downloaded third-party software packages are stored in current script directory by default.

2. **Compile Operator Package**

    Upload downloaded third-party software to offline environment. Can be stored in `third_party` directory or custom directory. **Recommend the former, its compilation command is consistent with command in online compilation scenario.**

    - **third\_party directory** (Recommended)

        Please create `third_party` directory in project root directory (if exists, no need to create). Copy third-party software to this specified directory. At this time compilation command is consistent with online compilation command. Please refer to [Online Compilation](#online-compilation).

    - **Custom Directory**

        Create `${cann_3rd_lib_path}` directory at any location in offline environment. Copy third-party software to this directory. Please ensure this directory has access permission.

        ```bash
        mkdir -p ${cann_3rd_lib_path}
        ```

        At this time compilation command needs to additionally add `--cann_3rd_lib_path=${cann_3rd_lib_path}` on top of online compilation command to specify third-party software location. Assuming storage path is `/path/cann_3rd_lib_path`, commands corresponding to different compilation methods are as follows:

        - Custom Operator Package

            ```bash
            bash build.sh --pkg --soc=${soc_version} [--vendor_name=${vendor_name}] [--ops=${op_list}] --cann_3rd_lib_path=${cann_3rd_lib_path}
            # Taking Abs operator compilation as example
            # bash build.sh --pkg --soc=ascend910b --ops=abs -j16 --cann_3rd_lib_path=/path/cann_3rd_lib_path
            ```

        - ops-math Full Package

            ```bash
            bash build.sh --pkg --soc=${soc_version} --cann_3rd_lib_path=${cann_3rd_lib_path}
            # bash build.sh --pkg --soc=ascend910b --cann_3rd_lib_path=/path/cann_3rd_lib_path
            ```

        - ops-math Static Library

            ```bash
            bash build.sh --pkg --static --soc=${soc_version} --cann_3rd_lib_path=${cann_3rd_lib_path}
            # bash build.sh --pkg --static --soc=ascend910b --cann_3rd_lib_path=/path/cann_3rd_lib_path
            ```

3. **Install/Uninstall Operator Package**

    Operator package results compiled in offline and online scenarios are the same, stored in build_out directory under project root directory by default. Installation and uninstallation operation commands are also the same. Please refer to [Online Compilation](#online-compilation).

## Local Verification

After source code package deployment, can execute UT test cases through build.sh in project root directory to verify whether project functions are normal.

> Note: Executing UT test cases depends on googletest unit testing framework. For detailed introduction, please refer to [googletest official website](https://google.github.io/googletest/advanced.html#running-a-subset-of-the-tests).

```bash
# Install test related requirements.txt dependencies under root directory
pip3 install -r tests/requirements.txt
# Method 1: Compile and execute UT test cases for specified operator and corresponding function (choose one)
bash build.sh -u --[opapi|ophost|opkernel|opkernel_aicpu] --ops=abs
# Method 2: Compile and execute all UT test cases
# bash build.sh -u
# Method 3: Compile all UT test cases but do not execute
# bash build.sh -u --noexec
# Method 4: Compile and execute UT test cases for corresponding function (choose one)
# bash build.sh -u --[opapi|ophost|opkernel|opkernel_aicpu]
# Method 5: Compile UT test cases for corresponding function but do not execute (choose one)
# bash build.sh -u --noexec --[opapi|ophost|opkernel|opkernel_aicpu]
# Method 6: Can specify soc compilation when executing UT test cases
# bash build.sh -u --[opapi|ophost|opkernel|opkernel_aicpu] [--soc=${soc_version}]
```

Taking verifying whether ophost function is normal as example, execute the following command:

```bash
bash build.sh -u --ophost
```

After execution completes, following content appears, indicating successful execution.

```bash
Global Environment TearDown
[==========] ${n} tests from ${m} test suites ran. (${x} ms total)
[  PASSED  ] ${n} tests.
[100%] Built target math_op_host_ut
```

$\$\{n\}$ represents n test cases executed, $\$\{m\}$ represents m test items, $\$\{x\}$ represents time consumed executing test cases, unit is milliseconds.
